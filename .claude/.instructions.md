# コーディング規約

Claude Code がコードを生成・編集する際に従うべきルールを定義する。

## 名前空間（namespace）規則

- 命名規則: `Technopro.Othello.<レイヤー>.<サブレイヤー>` の形式に統一する
- 対応例:

| 旧 namespace | 新 namespace |
|---|---|
| `Othello.Core.Models` | `Technopro.Othello.Core.Models` |
| `Othello.Core.Rules` | `Technopro.Othello.Core.Rules` |
| `Othello.Core.Game` | `Technopro.Othello.Core.Game` |
| `Technopro.Othello.Core.AI` | `Technopro.Othello.Core.AI` |
| `Othello.WPF.ViewModels` | `Technopro.Othello.WPF.ViewModels` |
| `Othello.WPF.Converters` | `Technopro.Othello.WPF.Converters` |
| `Othello.Tests.*` | `Technopro.Othello.Tests.*` |

- 各 .csproj には `<RootNamespace>Technopro.Othello.XXX</RootNamespace>` を設定する
- XAML の `x:Class` と `clr-namespace:` も同様に更新する

## 実装パターン

### async Task + CancellationToken パターン
`GameViewModel` の非同期メソッド（`ProcessAIMoveAsync`, `CheckAndProcessNextTurnAsync` 等）を追加・修正する際は以下のパターンを踏襲する。
新規ゲーム開始時に `_cts.Cancel()` が呼ばれ、`OperationCanceledException` で正常終了する。

```csharp
private async Task SomeAsyncMethodAsync(CancellationToken ct)
{
    try
    {
        await Task.Delay(..., ct);          // キャンセルで OperationCanceledException
        ct.ThrowIfCancellationRequested();  // 長時間処理の後に明示チェック
        // ... 処理
    }
    catch (OperationCanceledException)
    {
        // 新規ゲーム開始による正常なキャンセル。状態は StartNewGame が管理する。
    }
    catch (Exception ex)
    {
        StatusMessage = $"エラー: {ex.Message}";
    }
    finally
    {
        if (!ct.IsCancellationRequested) IsAIThinking = false;
    }
}
```

呼び出し側（fire-and-forget）:
```csharp
_ = SomeAsyncMethodAsync(_cts!.Token);
```

### 拡張メソッド
列挙型に機能を追加するときは、同じファイル内の `XxxExtensions` static クラスに拡張メソッドとして定義する（例: `PlayerColorExtensions`, `GameStateExtensions`, `DifficultyLevelExtensions`）。

### MoveResult の生成
`MoveResult` は直接コンストラクタを呼ばず、ファクトリメソッドを使う。

```csharp
MoveResult.Success("メッセージ", flippedPieces); // 成功
MoveResult.Failure("エラー理由");                 // 失敗
```

## コメントを書かない条件（共通）

以下の場合はコメントを書かない。過剰なコメントは読み手の負担になりコードの変更に追随できなくなる。

- **コードを読めば自明な内容**（変数名・メソッド名・型で意図が分かるもの）
- **処理の「何を」を繰り返すだけのコメント**（`// i をインクリメントする` など）
- **一時的なメモや作業ログ**（完了後に削除し、コードに残さない）

コメントを書くべき場面:
- **なぜ（Why）その実装になったか** — 制約・トレードオフ・特定バグの回避策など
- **非自明な前提・不変条件** — 読み手が驚くような挙動の根拠
- **パフォーマンス最適化** — 意図的に複雑にした理由

## C# コメント規約

**public / internal メンバー（必須）**
- **クラス・インターフェース・列挙型**: `/// <summary>` で概要を 1〜2 文で記述する
- **メソッド・プロパティ**: `/// <summary>` + `/// <param name="...">` + `/// <returns>` を記述する
- **テストメソッド**: `/// <summary>` に「何をすると何になるか（パス条件）」を明記する

**例外（doc コメント任意）**
- **自明な自動プロパティ**: 名前と型だけで意図が完全に分かる場合（`public int BlackScore { get; }`、`public string StatusMessage { get; set; }` 等）
- **インターフェース実装の定型メソッド**: `IValueConverter.Convert/ConvertBack`、`ICommand.CanExecute/Execute` などフレームワーク制約で実装するメソッド
- **XAML code-behind のイベントハンドラ・コンストラクタ**: UI フレームワークが要求する定型コード

```csharp
/// <summary>ボードの指定位置へ石を置き、結果を返す。</summary>
/// <param name="position">配置先の座標（0-7, 0-7）</param>
/// <returns>成功/失敗と反転した石のリストを含む結果</returns>
public MoveResult MakeMove(Position position) { ... }

/// <summary>
/// 有効な手を打つと着手成功かつターンが相手に移ることを確認する。
/// パス条件: MakeMove の戻り値が IsSuccess = true、かつ GameState が WhiteTurn に変わること。
/// </summary>
[Fact]
public void MakeMove_ValidMove_ChangesGameState() { ... }
```

**private メンバー（任意）**
- 処理が自明な private メソッドにはコメント不要
- 非自明なアルゴリズム・副作用・制約がある場合のみ `/// <summary>` またはインラインコメントを付ける

**インラインコメント（共通）**
- 条件分岐（`if` / `switch`）: 判定理由が名前から読み取れない場合のみ `//` で意味を補足する
- 重要な処理: アルゴリズムの意図・注意点・回避している既知バグを記述する

## Python コメント規約

**モジュール・クラス・関数（必須）**
- docstring（`"""..."""`）で概要・引数・戻り値を記述する
- 1 行で収まる場合は 1 行 docstring でよい

```python
def get_flips(board, r, c, player):
    """指定位置に stone を置いた際に反転される相手石の座標リストを返す。"""
```

**インラインコメント（任意）**
- 条件の意味が名前から分からない場合のみ `#` で補足する
- アルゴリズムの意図・前提条件・制約を記述する

---

## Rust コメント規約

**pub 関数・構造体・列挙型（必須）**
- `///` doc コメントで概要を記述する（`cargo doc` に反映される）

```rust
/// 指定位置への着手で反転する石の座標リストを返す。
/// 有効手でない場合は空のリストを返す。
pub fn get_flippable_pieces(board: &Board, row: usize, col: usize, player: i8) -> Vec<(usize, usize)> { ... }
```

**非 pub 関数（任意）**
- 処理が自明な場合はコメント不要
- 非自明なロジック・最適化・前提条件がある場合のみ `//` または `///` を付ける

**インラインコメント（任意）**
- `// SAFETY:` — unsafe ブロックがある場合は安全性の根拠を必ず記述する
- アルゴリズムの意図・枝刈り条件の根拠など非自明な部分に絞って記述する
